git --version
Verify Installation

Check if Git is installed correctly and display the installed version.

git --version
git config
Set Identity

Configure your identity for commits. Required before first commit. Use --global for system-wide settings or --local for project-specific.

git config --global user.name "Your Name"
git config --global user.email "your@email.com"
git config --global user.name "John Doe"
git init
Initialize Repository

Initialize a new Git repository in the current folder. Creates a hidden .git folder that tracks everything.

git init
Local Only
git clone
Copy Remote Repository

Download a remote repository to your local machine. Creates a local copy with full history.

git clone [repository-url]
git clone https://github.com/user/project.git

üìù Local Operations

git status
Check Repository Status

Shows which files are modified, staged, or untracked. Your go-to command to see what's happening in your working directory.

git status
git add
Stage Changes

Move changes from working directory to staging area. Review before committing!

git add filename.txt # Specific file
git add . # Current directory & subdirs
git add --all # Entire project (all changes)
git add -A # Same as --all
git add *.txt # All txt files (no deleted)
git add * # All files (no deleted)
git add index.html
Note: * excludes deleted files
git commit
Save Changes Permanently

Save staged changes to local repository with a descriptive message. Creates a snapshot you can return to.

git commit -m "Descriptive message"
git commit -m "Added navigation menu"
Creates History
git log
View Commit History

View commit history with IDs, authors, dates, and messages. Use --oneline for compact view.

git log # Detailed view
git log --oneline # Compact summary
git log --all --graph # Visual branch graph
git log --oneline
git diff
Compare Changes

Show differences between commits. Place newer commit ID first to see what's added/removed from that perspective.

git diff [newer-commit] [older-commit]
git diff a1b2c3d e4f5g6h
git rm
Remove Files

Delete files and stage the deletion in one step. Safer than manual deletion + git add.

git rm file.txt # Delete and stage
git rm -f file.txt # Force delete (ignore uncommitted changes)
git rm --cached file.txt # Remove from staging only (keep file)
git rm -r folder/ # Remove directory recursively
git reset
Undo Staging/Commits

Unstage files or undo commits. Different modes affect working directory differently.

git reset # Unstage files (keep changes)
git reset --hard # Discard all changes (dangerous!)
git reset HEAD~1 # Undo last commit (keep changes staged)
--hard deletes files permanently

From the transcript, here are the crucial differences between git add options:

  • git add --all or git add -A: Stages all changes across the entire project including modifications, new files, and deletions. Works from any directory.
  • git add .: Stages changes in the current directory and subdirectories only. If you're in a subfolder, it won't stage changes in the root.
  • git add *: Stages all visible changes except deleted files. Also limited to current directory scope.
  • git add *.txt: Stages all .txt files in current directory (not subdirectories) and excludes deletions.

Best Practice: Go to root directory and use git add . for simplicity, or git add --all when you need to ensure everything including deletions is staged.

üåø Branching & Merging

üçΩÔ∏è The Restaurant Kitchen Analogy

The main branch is the main kitchen serving customers. When developing a new feature, you create a separate branch (a test kitchen) to experiment safely. Once perfected, you merge it back into the main kitchen. This keeps the main branch stable while allowing parallel development.

git branch
Manage Branches

List existing branches or create new ones. The asterisk (*) shows your current branch.

git branch # List branches
git branch new-name # Create new branch
git branch -d name # Delete branch
git checkout
Switch Branches/Commits

Switch between branches or restore files. Also used to view previous commits by ID (detached HEAD state).

git checkout branch-name # Switch to branch
git checkout commit-id # View old commit
git checkout main # Return to latest
git checkout development
git merge
Combine Branches

Integrate changes from one branch into another. Creates a merge commit. May require conflict resolution.

git merge [source-branch] -m "Message"
git merge development -m "Merging dev updates"
git stash
Temporary Storage

Save uncommitted changes temporarily to switch branches. Like a clipboard for your work-in-progress.

git stash # Save current changes
git stash pop # Restore and remove from stash
git stash apply # Restore but keep in stash
git stash list # View stashed items
git stash drop # Delete stash
Pop = Cut, Apply = Copy
git rebase
Linearize History

Reapply commits on top of another base tip. Creates cleaner, linear history than merging. Never use on public/shared branches!

git checkout feature
git rebase main
Rewrites commit IDs - Team coordination required
git revert
Safe Undo

Undo a specific commit by creating a new commit that reverses it. Safe for public repositories (doesn't delete history).

git revert [commit-id]
Safe for shared repos Different from reset!
git restore
Restore Files

Discard uncommitted changes and restore file to last committed state. For undoing work before committing.

git restore file.txt # Restore file
git restore . # Restore all
git restore --staged file # Unstage file

Conflicts occur when the same file section is modified differently in two branches.

  1. Git marks the file as conflicted and stops the merge
  2. Open the file - you'll see conflict markers: 
    <<<<<<< HEAD
Your changes here
=======
Their changes here
>>>>>>> branch-name
  3. Edit the file to keep desired changes, removing all markers
  4. Stage the resolved file: git add .
  5. Complete the merge: git commit -m "Resolved merge conflict"

‚òÅÔ∏è Remote Operations

git push
Upload to Remote

Send local commits to remote repository. Updates the cloud version with your local changes.

git push origin main # Push main branch
git push origin branch-name # Push specific branch
git push origin development
git pull
Fetch & Merge

Download changes from remote and immediately merge into current branch. Equivalent to git fetch + git merge.

git pull [remote] [branch]
git pull origin main
git fetch
Check Remote Changes

Download remote changes to local memory without merging. Preview updates before applying them to your working directory.

git fetch
Safe to run anytime Doesn't affect working files
git remote
Manage Connections

View and manage remote repository connections. "Origin" is the default name for your remote.

git remote -v # View remote URLs
git remote add origin url # Add new remote

git fetch: Downloads new data from remote but keeps it separate from your working files. Your local code stays unchanged. Use this to review changes before integrating them.

git pull: Downloads new data AND immediately merges it into your current branch. Your working files are updated instantly. Equivalent to running git fetch followed by git merge.

Analogy: Fetch is like checking your mailbox (you see what came in but haven't opened it). Pull is like checking the mailbox and immediately opening everything.

üöÄ Advanced Features

git reset --hard
Destructive Reset

Permanently discard all changes (staged and unstaged) and restore to specified commit. Cannot be undone!

git reset --hard [commit-id]
Data loss risk!
git checkout [commit-id]
Time Travel

View project at specific point in history. Creates "detached HEAD" state - read only mode for inspection.

git checkout [commit-hash]
git checkout a1b2c3d
Remember to return: git checkout main
git diff
Compare Versions

Show exact line-by-line differences. Red = removed, Green = added. Order matters: git diff NEW OLD shows perspective from new commit.

git diff [commit-1] [commit-2]

ü§ù GitHub Collaboration

A Pull Request is a request to merge your branch into another (usually main). It's the standard code review workflow:

  1. Push your feature branch to GitHub
  2. Go to Pull Requests tab ‚Üí New Pull Request
  3. Select Base (target branch, e.g., main) and Compare (your branch)
  4. Review the diff of changes
  5. Add title and description explaining changes
  6. Create PR and request review from teammates
  7. After approval, click "Merge Pull Request"

Benefits: Code review, discussion, CI checks, and protected main branch.

  • Commit often, push frequently
  • Write meaningful commit messages describing "why" not just "what"
  • Create feature branches for new work - never develop directly on main
  • Use git stash when you need to switch contexts quickly
  • Pull before you push to avoid conflicts
  • Use git revert for public branches, git reset for private/local cleanup
  • Never use git rebase on commits already pushed to shared branches
  • Review changes with git diff before committing
