A comprehensive, searchable reference guide to version control and collaborative development. Based on the complete Git & GitHub course by Sumit Saha.

Introduction

Git vs GitHub

Git is a distributed version control system that runs locally on your computer. GitHub is a cloud-based hosting service for Git repositories. Think of Git as the coffee (the tool) and GitHub as the coffee shop (the service).

Git tracks every change you make to your files‚Äîwhat changed, when it changed, who changed it, and where. It works with almost any file type: code, images, text, or videos. The power of Git lies in its ability to save different versions of your files, allowing you to roll back to any previous state instantly.

Created by Linus Torvalds (creator of Linux), Git has become an essential tool that stays useful throughout your entire career.

Core Concepts

The Three States

1
Working Directory
Where you edit files on your local machine. This is your active project folder where you write code, create files, and make changes.
2
Staging Area
An intermediate "mirror" where you prepare changes before committing. Like checking your appearance before leaving for a party‚Äîyou can still make adjustments here.
3
Repository
The permanent storage where committed changes live. The .git folder contains all version history and metadata.

Local vs Remote

  • Local Repository: Lives on your computer. Contains all files, history, and branches.
  • Remote Repository: Lives in the cloud (GitHub, GitLab, Bitbucket). Acts as a central hub for collaboration and backup.

Installation & Configuration

git --version
Verify Git installation and display installed version
git --version
git config
Set your identity for commits. Git requires this before allowing commits.
git config --global user.name "Your Name"
git config --global user.email "your@email.com"
--global : All repositories --local : Current repo only
Installation: Download from git-scm.com for Windows, use Homebrew on Mac (brew install git), or your package manager on Linux.

Repository Setup

git init
Initialize a new Git repository in the current directory. Creates a hidden .git folder.
git init
git clone
Copy a remote repository to your local machine. Creates a local repository with all history.
git clone <repository-url>
git status
Check which files are modified, staged, or untracked. Shows the current state of your working directory.
git status

Staging & Committing

The basic workflow: Working Directory ‚Üí git add ‚Üí Staging Area ‚Üí git commit ‚Üí Repository

git add
Move changes from working directory to staging area
git add filename.txt # Specific file
git add . # Current directory & subdirs
git add --all # All changes (including deletes)
git add -A # Same as --all
git add *.txt # All .txt files (no deletes)
. : Current dir only --all/-A : Entire project * : Pattern match
git reset
Remove files from staging area (unstage), but keep changes in working directory
git reset # Unstage all
git reset HEAD~1 # Undo last commit (keep changes)
git reset --hard # Discard all changes (dangerous)
--hard : Discard changes --soft : Keep changes staged
git commit
Permanently save staged changes to the local repository
git commit -m "Descriptive message"
-m : Message flag --amend : Edit last commit
git rm
Delete files and stage the deletion in one step
git rm file.txt # Delete and stage
git rm -f file.txt # Force delete (if modified)
git rm --cached file.txt # Remove from staging only
git rm -r folder/ # Delete directory recursively
Note: git add * stages new/modified files but not deletions. Use git add --all or git add -A to include deleted files.

Viewing History

git log
View commit history with IDs, authors, and messages
git log # Full history
git log --oneline # Compact view
git log --graph # Visual branch graph
Press q to exit the log view.
git diff
Show differences between commits, branches, or working directory
git diff # Working vs Staging
git diff --staged # Staging vs Last commit
git diff commit1 commit2 # Compare two commits

Branching

Branches are separate lines of development. The default branch is main (previously master).

git branch
List, create, or delete branches
git branch # List branches
git branch new-feature # Create branch
git branch -d branch-name # Delete merged branch
git branch -D branch-name # Force delete
git checkout
Switch between branches or restore files
git checkout branch-name # Switch to branch
git checkout -b new-branch # Create & switch
git checkout commit-id # View old commit (detached HEAD)
git switch
Modern alternative to checkout (Git 2.23+) for switching branches
git switch branch-name
git switch -c new-branch # Create and switch

Branching Strategy

New features should be developed in separate branches. Once complete and tested, merge them back into main. This keeps the main branch stable while allowing experimentation.

Merging & Conflicts

git merge
Combine changes from one branch into current branch
# Switch to target branch first
git checkout main
git merge feature-branch

Resolving Merge Conflicts

Conflicts occur when the same file is modified differently in both branches. Git marks the file with conflict markers:

<<<<<<< HEAD
Your changes here
=======
Their changes here
>>>>>>> branch-name
  1. Open the conflicted file(s)
  2. Edit to keep desired changes (remove markers)
  3. git add . to stage resolved files
  4. git commit -m "Resolved merge conflict"

Remote Operations

git push
Upload local commits to remote repository
git push origin main # Push main branch
git push origin branch-name # Push specific branch
git push -u origin main # Set upstream (first push)
git fetch
Download changes from remote without merging. Updates remote-tracking branches.
git fetch
git fetch origin
git pull
Fetch and merge remote changes in one command (fetch + merge)
git pull origin main
git pull # Default remote/branch
--rebase : Rebase instead of merge

Remote Workflow

Push: Local ‚Üí Remote | Fetch: Remote ‚Üí Local (repo only) | Pull: Remote ‚Üí Local ‚Üí Working Directory

Undoing Changes

git restore
Discard uncommitted changes in working directory
git restore file.txt # Discard file changes
git restore . # Discard all changes
git restore --staged file.txt # Unstage file
git revert
Undo a specific commit by creating a new "reverse" commit. Safe for public history.
git revert abc123
git revert HEAD # Undo last commit
Unlike reset, revert doesn't delete history‚Äîit's the safe way to undo.

Stashing

Temporarily save uncommitted changes without committing them, allowing you to switch branches or work on something else.

git stash
Save uncommitted changes to a temporary area
git stash # Stash with auto-message
git stash push -m "message" # Stash with description
git stash pop / apply
Restore stashed changes. Pop removes from stash; apply keeps it.
git stash pop # Restore and remove from stash
git stash apply # Restore but keep in stash
git stash list # View all stashes
git stash drop stash@{0} # Delete specific stash

Rebasing

Rebasing moves or combines a sequence of commits to a new base commit, creating a linear history.

git rebase
Reapply commits on top of another base tip
# While on feature branch:
git rebase main # Move feature commits on top of main
Warning: Never rebase commits that have been pushed to a public/shared branch. Rebasing rewrites commit IDs and will cause confusion for collaborators.

Pull Requests (GitHub)

A Pull Request (PR) is a request to merge changes from one branch into another, typically reviewed by team members before merging.

1
Push Your Branch
Push your feature branch to GitHub: git push origin feature-branch
2
Create PR
On GitHub, click "New Pull Request", select base (main) and compare (your branch) branches
3
Review
Team reviews changes, leaves comments, requests modifications if needed
4
Merge
Once approved, click "Merge Pull Request" to integrate changes into main

Command Quick Reference

Essential Daily Commands

Daily Workflow
git pull # Update local repo
git checkout -b feature # Create branch
# ... make changes ...
git add .
git commit -m "message"
git push origin feature
Emergency Undo
git restore . # Discard uncommitted changes
git reset --hard HEAD # Discard to last commit
git revert HEAD # Undo last commit (safe)
Branch Cleanup
git branch -d branch-name # Delete local branch
git push origin --delete branch-name # Delete remote

Glossary

Commit
A snapshot of your repository at a specific point in time, containing changes, author info, timestamp, and a unique ID.
Branch
An independent line of development. The default is usually 'main' or 'master'.
Merge
Combining changes from different branches into a single branch.
Merge Conflict
When Git can't automatically merge changes because the same part of a file was modified differently in both branches.
Staging Area (Index)
A temporary area where you prepare changes before committing them to the repository.
Origin
The default name Git gives to the remote repository you cloned from.
HEAD
A pointer to the current commit in your working directory, usually the latest commit of your current branch.
Clone
Creating a local copy of a remote repository, including all history and branches.
Fork
Creating a personal copy of someone else's repository on GitHub, allowing you to freely experiment without affecting the original.
Pull Request (PR)
A GitHub feature to propose changes, request review, and discuss code before merging branches.
Repository (Repo)
A storage location for your project, containing files, folders, and Git history.
Stash
A temporary storage for uncommitted changes, allowing you to switch branches without committing incomplete work.
