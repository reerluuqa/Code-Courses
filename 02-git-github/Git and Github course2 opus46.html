Git & GitHub Handbook

A comprehensive, interactive reference for mastering Git version control and GitHub collaboration — from core concepts to real-world workflows.

How to use this handbook

Navigate using the sidebar, search for any command or concept, and click code blocks to copy. Every section includes syntax examples and practical explanations.

What is Git?

Git is a powerful version control system that constantly tracks every change you make to your files — what changed, when it changed, who changed it, and where. It works with virtually any file type: code, text, images, and more.

The magic of Git lies in its ability to save different versions of your files. You can roll back to any previous version at any time, ensuring you never lose work or accidentally overwrite something important.

Key Benefits

  • Version History — Every change is recorded and retrievable
  • Collaboration — Multiple developers can work on the same project simultaneously
  • Branching — Experiment safely without affecting the main codebase
  • Backup — Cloud-hosted repositories protect against data loss
  • Career Longevity — Once learned, Git stays useful for your entire career
History

Git was created by Linus Torvalds, the same brilliant mind behind Linux. It's one of those rare tools that, once learned, stays useful for the rest of your career.

Git vs GitHub

Git and GitHub are connected but fundamentally different. Think of it this way: Git is the coffee and GitHub is the coffee shop where that coffee is served.

Aspect Git GitHub
What it is A version control tool A cloud hosting platform for Git repositories
Where it runs Locally on your computer In the cloud (remote server)
Purpose Track changes, manage versions Collaborate, share code, backup
Owned by Open source community Microsoft
Alternatives — GitLab, Bitbucket
Remember

GitHub isn't the only remote hosting option. GitLab and Bitbucket are also popular and trusted by developers worldwide. However, GitHub remains the most widely adopted platform.

Git Architecture

Git's architecture is divided into two major parts: local (your computer) and remote (the cloud). Understanding the flow between these areas is the foundation of the entire system.

Step 1
Working Directory
→
Step 2
Staging Area
→
Step 3
Local Repository
→
Step 4
Remote Repository

Working Directory

The folder where you are actively working on your project. You write code, create files, modify existing ones — all the action happens here.

Staging Area

A temporary middle ground between your working directory and the repository. When you finish changes, staging means saying: "My changes are ready for the next step." It's an intermediate checkpoint where you review before saving permanently.

Local Repository

A specific folder on your computer where Git stores all versions of your files and their complete change history. When you commit, you permanently save staged changes here. The .git hidden folder is the core of this repository.

Remote Repository

Lives on a cloud server like GitHub. Used to share code with others, access it from another computer, or keep it safely backed up. Think of it like Google Drive, but specifically for code.

The .git folder

When you initialize a repository, Git creates a hidden .git directory. This is the core of your project — it stores all internal data: changes, history, commits, and more. Everything Git does is based on data stored within this folder.

Getting Started

Installation & Setup

Installing Git

Visit the official Git website and download the version for your operating system:

  • Windows: Choose 32-bit or 64-bit installer, run it
  • macOS: Install via Homebrew or the official installer
  • Linux: Follow the distribution-specific instructions

Verify Installation

git --version

If installed correctly, you'll see a version number. If not, you'll get an error message.

Initial Configuration

Git requires your identity before you can make commits. This is a one-time setup:

git config --global user.name "Your Name" git config --global user.email "you@example.com"
Global vs Local Config

Use --global to apply settings to your entire computer. Use --local instead to apply settings only to a specific repository.

Terminal Options

  • macOS: Terminal app
  • Windows: Command Prompt, PowerShell, or Git Bash (recommended — feels like a Linux terminal)
  • Linux: Any terminal emulator
Core Commands

Init & Clone

There are two ways to create a Git repository: initializing one locally, or cloning one from a remote server.

git init

Creates a new Git repository in your current directory. This tells Git: "Start tracking everything in this folder."

# Navigate to your project folder cd ~/Desktop/my-project # Initialize Git git init

After running this, Git creates a hidden .git directory — the core of the repository. You can verify with ls -la.

git clone

Downloads (clones) a remote repository to your local machine, including all files and history.

# Clone a repository from GitHub git clone https://github.com/username/repository.git # Navigate into the cloned folder cd repository
Where to find the clone URL

On GitHub, open the repository and click the green Code button. Copy the HTTPS link provided.

Core Commands

Git Status

The git status command shows you the current state of your working directory and staging area. It tells you which files have been modified, which are staged, and which are untracked.

git status

What it reveals

  • Modified files — Existing files that have been changed
  • Untracked files — New files Git doesn't know about yet
  • Staged files — Files ready to be committed (under "Changes to be committed")
  • Tracked files — Files Git already knows about from previous commits
Use it often

Run git status frequently. It's your go-to command for understanding exactly what's happening in your repository at any given moment.

Core Commands

Staging with git add

The git add command moves changes from the working directory to the staging area — telling Git: "I want to keep these changes."

Variations

Command Behavior
git add --all Stages every change across the entire project (new, modified, deleted)
git add -A Same as --all — shorthand version
git add . Stages changes in the current directory and subdirectories only
git add * Stages new and modified files, but not deleted files
git add filename Stages a specific file only
git add *.txt Stages all files matching a pattern in the current directory

Examples

# Stage everything in the entire project git add --all git add -A # Stage changes in current directory and below git add . # Stage a specific file git add 1.txt # Stage a file inside a folder git add my-folder/3.txt # Stage all .txt files git add *.txt
Key difference: dot vs star

git add . stages everything (including deletions) within the current directory tree. git add * stages new and modified files but ignores deletions. For most use cases, git add . from the root is the best practice.

Unstaging with git reset

To move files back from the staging area to the working directory:

# Unstage all files git reset # Unstage a specific file git reset filename
Core Commands

Committing

Committing means permanently saving your staged changes to the local repository, locking them in as a recorded version of your project's history.

The party analogy

You don't walk straight out the door for a party. First, you stand in front of a mirror (staging area) — check your clothes, shoes, hair. You make adjustments. Once everything looks perfect, you leave (commit). The staging area gives you that intermediate checkpoint to review before finalizing.

# Commit staged changes with a message git commit -m "Your descriptive commit message"

Undoing a Commit

# Undo the last commit, return changes to working directory git reset HEAD~
Write good commit messages

A commit message should describe what was changed and why. Good messages make your project history easy to understand for you and your collaborators.

Core Commands

Removing Files (git rm)

Instead of manually deleting a file and then staging the deletion, git rm does both in a single command.

Command Behavior
git rm file.txt Deletes the file and stages the deletion
git rm -f file.txt Force delete — even if the file has uncommitted changes
git rm --cached file.txt Removes from staging/tracking but keeps the file in your working directory
git rm -r folder/ Recursively removes a folder and all its contents

Reset Options

# Unstage changes only (files remain deleted on disk) git reset # Restore everything: staged changes AND deleted files git reset --hard
Be careful with --hard

git reset --hard discards all uncommitted changes permanently. Use it only when you're certain you want to throw away your work.

Core Commands

Log & Diff

Viewing Commit History

# Full commit log git log # Compact one-line format git log --oneline

Each commit shows its commit ID (a unique hash), author, date, and commit message. The --oneline flag gives a clean, shortened view.

Exiting the log view

If the log is long, press Q on your keyboard to exit the log viewer.

Comparing Commits with git diff

See the exact differences between two commits — what lines were added and removed.

# Compare two commits (newer ID first, older ID second) git diff <newer-commit-id> <older-commit-id>

The output shows removed lines in red and added lines in green. Reversing the order of IDs reverses the perspective.

Branching & Merging

Branching

A branch is a separate line of development where you can work independently. The default branch is called main (formerly master).

The restaurant kitchen analogy

The main branch is the main kitchen where dishes are served to customers. To test a new recipe, you create a separate test kitchen (a new branch). Once the recipe is perfected, you bring it back to the main kitchen.

# List all branches (current branch has a *) git branch # Create a new branch git branch development # Switch to a branch git checkout development # Switch back to main git checkout main
Branch inheritance

When you create a new branch, it inherits the exact state of the branch you were on at that moment. It's an exact copy of the current branch.

Branch Isolation

Changes made in one branch are completely invisible when you switch to another. Git manages your file system seamlessly — when you switch branches, the files shown change to match that branch's state. You can have hundreds of branches, each with its own changes.

Always commit before switching

For changes to belong to a branch, you must commit them while on that branch. Uncommitted changes can cause conflicts when switching branches.

Checking Out Commits

You can also use checkout to visit a previous commit by its ID:

# Go to a specific commit (detached HEAD state) git checkout <commit-id> # Return to the latest state git checkout main
Branching & Merging

Merging

Merging means combining changes from two branches into one. First switch to the target branch (where you want the changes to go), then merge the source branch.

# Switch to the branch you want to merge INTO git checkout main # Merge another branch into the current one git merge development -m "Merging development into main"

Merge Workflow Example

  1. Create a feature on the development branch
  2. Switch to main: git checkout main
  3. Merge: git merge development -m "Merge message"
  4. The changes from development now appear in main
Branching & Merging

Merge Conflicts

A merge conflict occurs when the same part of the same file has been changed differently in two branches. Git can't decide which version to keep, so it flags the conflict for you to resolve manually.

How Conflicts Happen

  1. Branch A modifies line 5 of file.txt
  2. Branch B also modifies line 5 of file.txt — differently
  3. Attempting to merge produces: "Automatic merge failed; fix conflicts and then commit the result."

Resolving Conflicts

Open the conflicted file. Git marks the conflicting sections clearly:

<<<<<<< HEAD your changes from the current branch ======= changes from the other branch >>>>>>> other-branch
  1. Decide which version to keep (or combine both)
  2. Remove the conflict markers (<<<, ===, >>>)
  3. Save the file
  4. Stage and commit: git add . then git commit -m "Merge conflict resolved"
Branching & Merging

Rebase

Rebase is an alternative to merging that creates a clean, linear commit history. Instead of creating a merge commit, it replays your branch's commits on top of another branch.

# Switch to the branch you want to rebase git checkout feature # Rebase onto main git rebase main

How Rebase Works Behind the Scenes

  1. Identifies the most recent common commit between both branches
  2. Temporarily sets aside all commits in your feature branch after that point
  3. Applies all new commits from main into your feature branch
  4. Reapplies your saved feature commits on top, one by one

Merge vs Rebase

Aspect Merge Rebase
History Creates merge commits; non-linear Clean, linear history
Commit IDs Preserved Rewritten (new IDs)
Safety Safe for shared branches Risky on shared branches
Golden Rule of Rebase

Never rebase public/shared branches. Rebase rewrites commit history — including commit IDs. If others are working on the same branch, their copies won't match and they can't sync normally. Only rebase on local or personal branches where only you are working.

Remote Operations

Push, Fetch & Pull

git push

Sends your local commits to the remote repository.

# Push current branch to remote git push origin main # Push a different branch git push origin development # Push staging branch (creates it on remote if it doesn't exist) git push origin staging

git fetch

Downloads remote changes into your local repository's memory, but doesn't merge them into your working directory.

# Fetch remote changes git fetch # Then merge manually git merge

git pull

Combines fetch and merge in a single command. Immediately updates your working directory with remote changes.

# Fetch + merge in one step git pull
The relationship

git pull = git fetch + git merge. Use fetch when you want to review changes before merging. Use pull when you want everything updated immediately.

Command Direction Behavior
git push Local → Remote Sends commits to remote
git fetch Remote → Local (memory) Downloads but doesn't merge
git pull Remote → Local (files) Downloads and merges automatically
Remote Operations

Pull Requests

A Pull Request (PR) is a request you make on GitHub to merge your changes from one branch into another. It's a way of saying: "I've made some changes — please review and merge them."

Creating a Pull Request

  1. Push your branch to GitHub: git push origin development
  2. On GitHub, go to the Pull Requests tab
  3. Click New Pull Request
  4. Set the base branch (where changes go, e.g., main)
  5. Set the compare branch (where changes come from, e.g., development)
  6. Review the comparison — which files changed, what was added/removed
  7. Click Create Pull Request
  8. Add a title and description
  9. Submit the PR

PR Sections

  • Conversation — Discuss and comment on the changes
  • Commits — Lists all commits included in the PR
  • Files Changed — Shows exactly what changed in which files

Completing a Pull Request

After review, click Merge Pull Request → Confirm Merge. GitHub will display: "Pull request successfully merged and closed."

Why PRs matter

Pull requests ensure every change gets reviewed before merging into main. The main branch stays stable, and teams collaborate safely and efficiently. This is how professional software companies use Git and GitHub.

Undo & Recovery

Git Restore

The git restore command reverts files back to their last committed state. It's used to undo local, uncommitted changes.

# Restore a specific file to its last committed state git restore filename.txt # Restore an entire directory git restore directory-name/ # Restore everything in the working directory git restore . # Unstage a file (remove from staging, keep changes) git restore --staged filename.txt # Unstage everything git restore --staged .
Command Effect
git restore file Discards working directory changes, reverts to last commit
git restore --staged file Removes from staging area, keeps file changes intact
Undo & Recovery

Git Stash

Stash lets you temporarily set aside unfinished work, switch branches, then bring your changes back later. It's like putting your work in a safe drawer.

When to Use Stash

You're halfway through a feature, but need to switch branches to review something else. Your work isn't ready to commit, and Git won't let you switch with uncommitted changes. Solution: stash it.

# Save current changes to stash git stash # Switch branches freely git checkout other-branch # Come back and restore your work git checkout main git stash pop

Pop vs Apply

Command Restores Changes Removes from Stash List
git stash pop ✅ Yes ✅ Yes
git stash apply ✅ Yes ❌ No (stays in list)

Managing Multiple Stashes

# View all stashed items git stash list # Apply a specific stash git stash apply stash@{0} # Pop a specific stash git stash pop stash@{0} # Remove a specific stash git stash drop # Remove a specific stash by ID git stash drop stash@{0}
Stash order

Git keeps stashes in order: newest on top (stash@{0}), oldest at the bottom. Pop and apply without a specifier always act on the most recent stash.

Undo & Recovery

Reset & Revert

git reset

Takes you back to a specific commit and discards all commits after that point. No new commit is created.

# Unstage files (keep changes in working directory) git reset # Undo the last commit (keep changes in working directory) git reset HEAD~ # Hard reset: undo everything, restore deleted files git reset --hard

git revert

Undoes the changes from a specific commit by creating a new commit that reverses them. The project history stays clean and traceable.

# Revert a specific commit (creates a new reversal commit) git revert <commit-id>

The cooking analogy

If you accidentally added too much salt, revert is like adding extra ingredients to balance it — you fix the mistake without throwing the dish away. Reset is like starting the dish over from scratch.

Reset vs Revert

Aspect git reset git revert
Creates new commit? ❌ No ✅ Yes
History Removes commits from log Preserves all history
Safe for shared repos? ❌ No ✅ Yes
Use case Local undo Public/collaborative undo

Reference

Glossary

Key terms and concepts explained in plain language.

Repository (Repo)

A storage location where Git keeps all versions of your files and their complete change history. Can be local (on your machine) or remote (on GitHub).

Working Directory

The project folder on your computer where you actively create, edit, and modify files. This is where all the action happens.

Staging Area (Index)

A middle ground between the working directory and the repository. Files placed here are "ready to commit" — it's an intermediate checkpoint for review.

Commit

A permanent snapshot of your staged changes saved to the local repository. Each commit has a unique ID, author info, timestamp, and descriptive message.

Branch

A separate, independent line of development. Branches let you work on features or fixes without affecting the main codebase.

Main / Master

The default primary branch in a Git repository. This is the central line of development where stable, production-ready code lives.

Merge

The process of combining changes from one branch into another, integrating separate lines of development into a unified codebase.

Merge Conflict

Occurs when the same part of the same file has been changed differently in two branches. Git can't auto-resolve it, requiring manual intervention.

Rebase

An alternative to merging that replays your branch's commits on top of another branch, creating a clean, linear commit history.

Clone

Downloading a complete copy of a remote repository (including all files and history) to your local machine.

Push

Sending your local commits to a remote repository (e.g., uploading to GitHub).

Pull

Fetching remote changes and immediately merging them into your local branch. Equivalent to git fetch + git merge.

Fetch

Downloading remote changes into your local repository's memory without merging them into your working directory.

Pull Request (PR)

A GitHub feature that lets you request merging your branch's changes into another branch, enabling code review before integration.

Origin

The default name Git gives to the remote repository you cloned from. Used in commands like git push origin main.

HEAD

A pointer that refers to your current position in the repository — usually the latest commit on your current branch.

Stash

A temporary storage area where you can save uncommitted changes, allowing you to switch branches and return to your work later.

Version Control System (VCS)

A system that records changes to files over time so you can recall specific versions later. Git is the most popular VCS.

.git Folder

A hidden directory created by git init. Contains all internal data Git needs: change history, commits, branch info, and configuration.

Commit ID (Hash)

A unique identifier (SHA-1 hash) assigned to every commit. Used to reference, compare, or revert to specific commits.

Tracked vs Untracked Files

Tracked files are known to Git from previous commits. Untracked files are new and haven't been staged or committed yet.
