Core Concepts: Git vs GitHub
Fundamentals ‚ñº
Git and GitHub are connected but completely different. Git is the coffee ‚òï ‚Äî a powerful version control tool that runs locally on your machine. GitHub is the coffee shop üè™ ‚Äî a cloud platform where that code is hosted and shared.

‚òï What is Git?

Git is a version control system (VCS) that constantly tracks every change you make to your files ‚Äî what changed, when, who changed it, and where. It works with virtually any file type: code, text, images, and more. Git saves different versions of your files so you can roll back to any previous version at any time. Created by Linus Torvalds (the creator of Linux), Git is one of those rare tools that remains useful for your entire career.

üêô What is GitHub?

GitHub is a cloud-based hosting platform for Git repositories. It acts as a central online server where your team's entire project lives, making it easy for everyone to see, edit, and share updates. Other alternatives include GitLab and Bitbucket, but GitHub (now owned by Microsoft) is the most popular.

üîÑ Real-Life Example

You spend hours coding a project and the client loves it. A month later, the client requests changes. You update the code. Then the client says: "Actually, the old version was better." Without Git, you've overwritten the original ‚Äî it's gone. With Git, every version is saved. You simply roll back.

Core ‚ñº
Git's architecture is divided into two major parts: Local (your computer) and Remote (the cloud, usually GitHub). Understanding the flow between the three local stages ‚Äî Working Directory ‚Üí Staging Area ‚Üí Repository ‚Äî is the foundation of everything in Git.

üìÇ Working Directory

The folder on your computer where all the action happens. You write code, create files, modify existing ones, and make changes here.

üìã Staging Area

A temporary middle ground between your working directory and the repository. When your changes are ready, you stage them ‚Äî like standing in front of a mirror before a party, making sure everything looks right before you go out.

üóÑÔ∏è Local Repository

A specific folder on your computer (the hidden .git directory) where Git permanently stores every committed version and the complete change history. Git creates system files here to track changes, history, and commits.

‚òÅÔ∏è Remote Repository

Lives on a cloud server (GitHub, GitLab, Bitbucket). You push local changes here to share and back up your code ‚Äî just like using Google Drive for files. If needed, you pull remote changes back to your machine.

‚ÑπÔ∏è The .git hidden folder is the core of your repository. It tracks all changes, history, commits, and internal data. To see it, use ls -la in the terminal.

Git Configuration
Required ‚ñº
Before your first commit, Git needs to know who you are. This identity is attached to every commit in the project history, helping track who made which changes.
üìß Set Email (Global)

Sets your email address globally for all repositories on this machine. This email appears in every commit you make.
basic
üë§ Set Name (Global)

Sets your display name globally. This name is attached to every commit.
basic
üìÅ Set Config (Local)

Use --local instead of --global to apply settings only to the current repository. Useful when using different identities for different projects.
intermediate
‚ö†Ô∏è If you see "Please tell me who you are" when trying to commit, just run the two config commands above. This is a one-time setup per machine (with --global).

Staging Changes ‚Äî git add
8 Commands ‚ñº
Moving changes from the Working Directory to the Staging Area is called adding. The staging area is like standing in front of a mirror before a party ‚Äî you review everything before finalizing.
üì¶ Stage Everything (--all)
git add --all
Stages every single change across the entire project ‚Äî new, modified, and deleted files. Works regardless of your current directory location.
basic
üì¶ Stage Everything (-A)
git add -A
Shorthand for git add --all. Behaves identically ‚Äî stages all changes project-wide.
basic
üìÅ Stage Current Directory (.)
git add .
Stages changes only in the current directory and its subdirectories. If you're in the root folder, it behaves like --all. If in a subfolder, only that subfolder's changes are staged.
basic
‚≠ê Stage with Wildcard (*)
git add *
Stages new and modified files but NOT deleted files. This is the key difference from . and --all. Only stages visible changes in the current directory (not subfolders).
intermediate
üìÑ Stage a Specific File
git add filename.txt
Stages only the named file. Use git add folder/file.txt for files inside subdirectories.
basic
üìé Stage by Extension
git add *.txt
Stages all files with the specified extension (e.g., .txt) in the current directory. Excludes deleted files and files in subfolders.
intermediate
‚è™ Unstage All Changes
git reset
Removes all files from the staging area and returns them to the working directory. Changes are preserved ‚Äî only the staging is undone.
basic
üí• Hard Reset
git reset --hard
Restores everything ‚Äî both staged changes AND deleted files ‚Äî back to the last committed state. A normal git reset only unstages; --hard fully reverts the working directory too.

Committing Changes
Essential ‚ñº
Committing means permanently saving staged changes to your local repository. Each commit is a recorded snapshot with a unique ID, a message, and the author's identity. Think of it as leaving for the party after you've checked everything in the mirror.
üíæ Make a Commit
git commit -m "Your commit message"
Saves staged changes permanently to the local repository. The -m flag lets you add a short description. The terminal shows how many files changed, lines added/deleted.
basic
‚è™ Undo Last Commit
git reset HEAD~1
Undoes the most recent commit and moves all changes back to the working directory. You can then modify or re-commit as needed. Complete control over your work!
intermediate
‚úèÔ∏è Commit messages matter! Write clear, descriptive messages like "Add login validation" or "Fix navbar alignment". They help you and your team understand what each commit does.

Removing Files ‚Äî git rm
5 Commands ‚ñº
Instead of manually deleting files and then staging the deletion, git rm does both in one step ‚Äî deletes the file AND stages the change.
üóëÔ∏è Remove & Stage
git rm filename.txt
Deletes the file from the working directory AND automatically stages the deletion. No need for a separate git add step.
basic
üí™ Force Remove

Force-deletes a file even if it has uncommitted local modifications. Without -f, Git protects you by refusing to delete modified files.
intermediate

Removes the file from the staging area / tracking, but keeps the actual file in your working directory. The file becomes "untracked." Useful for files you accidentally added.
intermediate
üìÅ Remove Folder Recursively
git rm -r foldername
Deletes a folder and all its contents (subfolders and files) recursively, and stages all deletions. Without -r, only the folder itself is removed, not its contents.

Log & Diff ‚Äî Viewing History
Inspection ‚ñº
View your complete commit history with git log and compare changes between commits with git diff. Each commit has a unique ID that lets you navigate to any point in history.
üìú Full Commit Log
git log
Shows the full commit history with commit IDs, author, date, and messages. Older commits appear at the bottom. Press Q to exit the log view.
basic
üìã Compact Log

Shows a clean, compact summary ‚Äî one commit per line with shortened commit IDs. These short IDs can also be used for checkout and other operations.
basic
üîç Compare Commits
git diff <newer-id> <older-id>
Shows exact differences between two commits: which files changed, lines removed (red) and lines added (green). Place the newer commit ID first for forward comparison.
intermediate
‚è∞ Checkout Previous Commit
git checkout <commit-id>
Switches your working directory to the exact state of that commit. Terminal shows "HEAD detached". Ensure all current changes are committed first. Use git checkout main to return to the latest state.

Branching
Core Feature ‚ñº
A branch is a separate line of development. Think of it like a restaurant: the main branch is the main kitchen serving customers. When you want to try a new recipe, you create a test kitchen (a new branch). Once the recipe is perfected, you bring it back to the main kitchen.

üç≥ Why Use Branches?

Instead of making changes directly in the main branch (risking breakage), you create a separate branch to develop and test. Once stable, you merge it back. This keeps your main project safe while allowing parallel development.

üìã List Branches
git branch
Shows all local branches. The current branch has a ‚òÖ star next to it. Default branch is main (formerly master).
basic
üå± Create New Branch
git branch branch-name
Creates a new branch that inherits the exact state of the branch you're currently on. You remain on the current branch after creation.
basic
üîÄ Switch Branch
git checkout branch-name
Switches to the specified branch. Git instantly adjusts your file system to show only the files relevant to that branch. All uncommitted changes must be committed or stashed first.
basic
üîë Key insight: When you switch branches, Git hides/shows files automatically. Files created in one branch are invisible from another until merged. Commits belong exclusively to the branch they were made in.

Merging & Conflict Resolution
Collaboration ‚ñº
Merging means combining changes from two branches into one. You can create multiple branches (staging, development, front-end, back-end), work on each separately, and merge them back into main once finalized.
üîó Merge a Branch
git merge branch-name -m "Merge message"
Merges the specified branch into your current branch. First switch to the target branch (e.g., main), then merge from the source (e.g., development). A merge commit is created.
basic collaboration

üí• Merge Conflicts

A conflict occurs when the same part of the same file has been changed differently in two branches. Git can't decide which version to keep, so it flags the file and asks you to resolve it manually.

üîß Resolving Merge Conflicts ‚Äî Step by Step

1. Git shows: "Automatic merge failed; fix conflicts and commit the result."
2. Open the conflicting file in a text editor. Git marks the conflicting sections with markers.
3. Decide which changes to keep ‚Äî one version, the other, or both combined.
4. Remove all conflict markers, save the file.
5. Run git add . then git commit -m "Merge conflict solved".

‚ö†Ô∏è Conflicts only happen when the same lines in the same file are modified in both branches. Changes to different files or different lines merge automatically without issues.

Restoring Changes ‚Äî git restore
Undo ‚ñº
git restore reverts files back to their last committed state. It's used to undo local, uncommitted changes ‚Äî whether in the working directory or the staging area.
üìÑ Restore a Single File
git restore filename.txt
Instantly reverts the file to its last committed version. All uncommitted modifications are discarded permanently.
basic
üîÑ Restore Everything
git restore .
Restores ALL files across the entire repository to their last committed state. Discards every uncommitted change.
basic
üìã Unstage Files
git restore --staged filename.txt
Removes files from the staging area but keeps the changes in the working directory. Use git restore --staged . to unstage everything.
intermediate
üìÅ Restore a Directory
git restore dirname/
Reverts all files within the specified directory to their last committed state.
basic
‚ö†Ô∏è git restore permanently discards uncommitted changes. Make sure you really want to undo before running it ‚Äî there's no way to recover discarded changes.

Working on a feature but need to switch branches? git stash temporarily saves your unfinished work, lets you switch branches freely, and brings your changes back when you return. Like putting your half-finished puzzle in a box to clear the table, then taking it back out later.

Restores stashed changes to your working directory but keeps them in the stash list for future reuse. Like copy + paste.

Git Revert ‚Äî Safe Undo
Safe History ‚ñº
git revert undoes a previous commit by creating a new commit that reverses those changes. It doesn't delete any history ‚Äî it corrects mistakes while keeping the timeline clean and traceable. Like adding extra ingredients to balance too much salt instead of throwing the whole dish away.
‚èÆÔ∏è Revert a Commit
git revert <commit-id>
Creates a new commit that undoes all changes from the specified commit. The original commit remains in the history. A prompt appears for the commit message ‚Äî type :wq (write & quit) to save the default message and exit.

Git Rebase ‚Äî Clean History
Advanced ‚ñº
git rebase re-applies your branch's commits on top of another branch, creating a clean, linear history without extra merge commits. Powerful but must be used carefully.

üîÑ How Rebase Works (Behind the Scenes)

1. Git finds the most recent common commit between your branch and the target branch.
2. All commits in your branch after that point are temporarily set aside.
3. The new commits from the target branch are applied to your branch.
4. Your saved commits are reapplied on top, one by one.
Result: A clean, linear commit history ‚Äî much tidier than merging.

üîÑ Rebase onto Main
git rebase main
Run from your feature branch to bring in all new commits from main while keeping a linear history. Your feature branch commits are reapplied on top of main's latest commits.

Pull Requests (PRs) ‚Äî GitHub Collaboration
GitHub ‚ñº
A Pull Request (PR) is a request to merge your changes into another branch (usually main). It's how you say: "I've finished my work ‚Äî please review it and merge it." This ensures every change gets reviewed before becoming part of the main codebase.

üìù Creating a Pull Request ‚Äî Step by Step

1. Push your branch to GitHub: git push origin branch-name
2. Go to your repository on GitHub.
3. Click the Pull Requests tab ‚Üí New Pull Request.
4. Set base = target branch (e.g., main) and compare = your branch (e.g., development).
5. GitHub shows a detailed comparison of changes.
6. Add a title and description ‚Üí Create Pull Request.
7. Team reviews the code in three sections: Conversation, Commits, Files Changed.
8. If approved ‚Üí Merge Pull Request ‚Üí Confirm Merge.
9. Success: "Pull request successfully merged and closed."

üëÄ PR Review Sections

üí¨ Conversation: Discuss and comment on changes with teammates.
üìù Commits: See all commits included in the PR.
üìÇ Files Changed: View exactly what changed in which files ‚Äî line by line.

‚úÖ Pull requests ensure the main branch stays stable. Every change is reviewed first, then merged. This is how professional software teams use Git and GitHub to collaborate safely and efficiently.
