The Call Stack

JavaScript is single-threaded with one call stack. It can only do one thing at a time. Understand LIFO (Last In, First Out) execution.

‚è∞

Macrotasks (Task Queue)

setTimeout, setInterval, and I/O operations go here. Executed one per loop iteration.

‚ö°

Microtasks

Promises, queueMicrotask, and MutationObserver. Higher priority than macrotasks. The entire queue is drained before next render.

üé®

requestAnimationFrame

Syncs with the browser's refresh rate (usually 60fps). Runs before repaint. Ideal for animations but can be paused in background tabs.

How the Event Loop Works

The event loop is the secret behind JavaScript's asynchronous programming. Despite being single-threaded, JS can perform non-blocking operations.

The Algorithm (Simplified)

  1. Execute the current Call Stack until it's empty
  2. Move all Microtasks from queue to stack and execute (repeat until microtask queue empty)
  3. Execute requestAnimationFrame callbacks
  4. Render/Update the UI (if needed)
  5. Execute one Macrotask from the Task Queue
  6. Return to step 1

üéØ Key Insight

Microtasks can starve the browser! If you keep adding microtasks recursively, macrotasks (like setTimeout) and rendering will never get a chance to run. This is a common interview trap.

Common Mistakes & Pitfalls

Avoid these traps that even experienced developers fall into.

Mistake #1: Assuming setTimeout(0) is Immediate

setTimeout with 0ms doesn't execute immediately. It goes to the macrotask queue, which only gets processed after microtasks and the current stack clear.

‚ùå Wrong Assumption
// Assuming this order: A, B, C console.log('A'); setTimeout(() => console.log('B'), 0); Promise.resolve().then(() => console.log('C')); // Reality: A, C, B
‚úÖ Correct Understanding
// Microtasks (Promises) always run // before Macrotasks (setTimeout) console.log('A'); // Sync Promise.resolve().then(() => { console.log('C'); // Microtask }); setTimeout(() => { console.log('B'); // Macrotask }, 0);
Mistake #2: Blocking the Main Thread

Heavy computation freezes the UI because JavaScript can't render while executing code on the stack.

‚ùå Blocking
// UI freezes for 5 seconds! const bigArray = new Array(10000000); for (let i = 0; i < bigArray.length; i++) { heavyCalculation(bigArray[i]); }
‚úÖ Yielding to Event Loop
// Process in chunks function processChunk(i) { if (i >= bigArray.length) return; // Process 1000 items at a time for (let j = 0; j < 1000 && i < bigArray.length; j++, i++) { heavyCalculation(bigArray[i]); } // Yield control setTimeout(() => processChunk(i), 0); } processChunk(0);
Mistake #3: Microtask Starvation

Recursively creating microtasks can block the event loop indefinitely, preventing renders and macrotasks.

function starveEventLoop() { Promise.resolve().then(() => { console.log('Microtask'); starveEventLoop(); // Recursive microtask! }); } // This will prevent ANY setTimeout from running // and freeze the UI rendering!
Fix: Use setTimeout(fn, 0) instead if you need to yield between iterations, or use requestAnimationFrame for visual updates.
Mistake #4: Confusing RAF with setTimeout

requestAnimationFrame pauses in background tabs to save battery, while setTimeout continues running.

// In a background tab: setTimeout(updateGame, 16); // Keeps running! üíªüî• requestAnimationFrame(updateGame); // Pauses! ‚úÖ

Use RAF for visual updates, setTimeout for logic that must continue when tab is hidden.

üéØ Interview Tip

When asked about Event Loop questions, always mention:

  • Synchronous code executes first (Call Stack)
  • Then all Microtasks (Promises) are drained
  • Then one Macrotask runs
  • Then RAF (if render needed)
  • Then the cycle repeats
